(Cwe-352)
The web application does not, or can not, sufficiently verify 
whether a well-formed, valid, consistent request was intentionally provided by the user who submitted the request
When a web server is designed to receive a request from a client without any mechanism for verifying that it was intentionally sent
 then it might be possible for an attacker to trick a client into making an unintentional request to
 the web server, which will be treated as an authentic request
 This can be done via a URL
image load, XML,
HttpRequest and can result in exposure to data or unintended code execution
This is normally during architecture and design  but can occur after depending on situations.
 The [ capsec-62] definition 
An attacker crafts malicious 
web links and distributes them 
via web pages, email, typically 
in a targeted manner
hoping to induce users to click on the link and execute malicious action against some party or application.
 If successful, the action embedded in the malicious link
 will be processed and accepted by the targeted application with the 
users' privilege level. 
This type of attack leverages the persistence and implicit trust placed in user session cookies by many web applications today.
In such an architecture, once 
the user authenticates to an application, and a session cookie is created on the user's 
system, all following transactions for that session are authenticated using that cookie, including potential actions initiated by an attacker and simply "riding" 
existing session cookie.

Possible  mitigation Use the
 "double-submitted cookie" method as described by
 Felten and Zeller: When a user visits a site, the site should generate a pseudorandom value and set it as a cookie on the
 user's machine. The site should require every form submission to include this value as a form value and also as a cookie value. When a POST request is sent to the site, the request should only be considered valid if the form value and the cookie value are the same.
Because of the
 same-origin policy, an attacker cannot read or modify the value stored in the cookie. To successfully submit a form on behalf 
of the user, the attacker would have to correctly guess the pseudorandom value. If the pseudorandom value is cryptographically strong, this would be difficult.This technique requires Javascript, so it may not work for browsers that have Javascript disabled
